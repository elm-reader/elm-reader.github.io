The Elm Reader
An exceptionally powerful debugging environment and learning tool

The Elm Reader is an extension to the debugger of the Elm programming language.

Leveraging Elm's pure functional design, the Elm Reader allows complete time-traveling debugging: exploration of the full trace of a program as a lazily-evaluated tree. Stepping backward is as easy as stepping forward.

The Future

Fine-grained time-traveling debugging opens up design opportunities for all kinds of developer tooling. There is much more to be done!
  • "Bookmarks" not "breakpoints" — When using traditional debuggers, we often have to stop and think to decide whether to Continue to the next breakpoint. If we go to far, we'll miss the anomaly and have to start over. Time-traveling debugging radically changes this, allowing the developer to freely navigate between and around every "breakpoint" or "bookmark" in an execution, without loss of analytical power or performance.
  • Examples in documentation — Examples in documentation can turbocharge onboarding. But they generally don't help you understand what's happening beneath the exact API used in the example. The Elm Reader is a way to improve the teaching potential of code examples, allowing dynamic exploration of how any invoked function works. Ultimately we learn by poking and experimenting, and time-traveling debugging makes it easy to explore a program execution with flexibility.
  • "Where did that state go wrong?" — Often when debugging, one knows some state to be incorrect and the whole problem is to identify when and where that state was created. In the Elm Reader, it is feasible to add a "find origin" tool to record a runtime ID for a value in a stack frame, re-run the tick in the program, and rewind to any frame which used that value, in particular the one which created it. To be sure, this incurs a runtime cost, but a manageable one for the great majority of cases.
  • Prefetching information — The reader already "prefetches" the values of each expression in a stack frame for immediate inspection. It's trivial to supplement this with the execution time of each invoked function and other useful metadata while imposing trivial runtime cost due to the lazy computation of trace information at runtime. Effectively, that metadata can be computed on just the frame being viewed, with all sub-frames (and hence most of the computation) running untraced.
